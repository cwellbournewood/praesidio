# Supply-chain security

Every Section release is built, signed, and attested in GitHub Actions
using OIDC-issued short-lived credentials — no long-lived signing keys
exist in the project. This document explains what gets produced and how
to verify it.

## What we publish per release

| Artefact | Where | Signed by |
|---|---|---|
| `ghcr.io/cwellbournewood/gateway:<version>` (linux/amd64, linux/arm64) | GitHub Container Registry | cosign keyless (Sigstore) |
| `ghcr.io/cwellbournewood/ui:<version>` | GitHub Container Registry | cosign keyless (Sigstore) |
| CycloneDX SBOM per image | release asset + cosign-attested in-registry | cosign attest |
| SLSA-3 build provenance per image | in-registry attestation | slsa-github-generator |
| `oci://ghcr.io/cwellbournewood/charts/section:<version>` Helm chart | GitHub Container Registry | cosign keyless |
| `section-<version>.tgz` Helm chart tarball | release asset | cosign sign-blob (with the checksum file) |
| `section-release.sha256` checksum manifest | release asset | cosign sign-blob (`.sig` + `.pem`) |

The release workflow that produces all of these is
`.github/workflows/release.yml`. It runs only on tags that match
`v*.*.*`, never on branch pushes.

## Verifying images

Cosign keyless signatures embed the GitHub Actions OIDC identity that
performed the sign. Verify against the expected identity:

```bash
COSIGN_EXPERIMENTAL=1 cosign verify \
    --certificate-identity-regexp '^https://github\.com/cwellbournewood/section/\.github/workflows/release\.yml@refs/tags/v.*' \
    --certificate-oidc-issuer https://token.actions.githubusercontent.com \
    ghcr.io/cwellbournewood/gateway:1.0.0 | jq .
```

You can pin to a digest for full immutability:

```bash
cosign verify \
    --certificate-identity-regexp '^https://github\.com/cwellbournewood/section/\.github/workflows/release\.yml@refs/tags/v.*' \
    --certificate-oidc-issuer https://token.actions.githubusercontent.com \
    ghcr.io/cwellbournewood/gateway@sha256:<digest>
```

## Verifying the SBOM attestation

```bash
cosign verify-attestation \
    --type cyclonedx \
    --certificate-identity-regexp '^https://github\.com/cwellbournewood/section/\.github/workflows/release\.yml@refs/tags/v.*' \
    --certificate-oidc-issuer https://token.actions.githubusercontent.com \
    ghcr.io/cwellbournewood/gateway@sha256:<digest> | \
    jq -r '.payload' | base64 -d | jq .predicate
```

## Verifying the SLSA-3 provenance

```bash
slsa-verifier verify-image ghcr.io/cwellbournewood/gateway@sha256:<digest> \
    --source-uri github.com/cwellbournewood/section \
    --source-tag v1.0.0
```

`slsa-verifier` lives at https://github.com/slsa-framework/slsa-verifier.
The provenance is generated by the official
`slsa-framework/slsa-github-generator` workflow and includes a
machine-verifiable record of:

* the source repository and commit SHA,
* the builder identity (GitHub Actions),
* the trigger event (tag push),
* the materials (base images, build args).

## Verifying the Helm chart

The chart is pushed to `ghcr.io/cwellbournewood/charts/section` as an OCI
artefact and signed by the same workflow identity:

```bash
cosign verify \
    --certificate-identity-regexp '^https://github\.com/cwellbournewood/section/\.github/workflows/release\.yml@refs/tags/v.*' \
    --certificate-oidc-issuer https://token.actions.githubusercontent.com \
    ghcr.io/cwellbournewood/charts/section:1.0.0
```

If you fetch the tarball from the GitHub Release page (not the OCI
registry), verify the checksum manifest signature first:

```bash
cosign verify-blob \
    --certificate section-release.sha256.pem \
    --signature section-release.sha256.sig \
    --certificate-identity-regexp '^https://github\.com/cwellbournewood/section/\.github/workflows/release\.yml@refs/tags/v.*' \
    --certificate-oidc-issuer https://token.actions.githubusercontent.com \
    section-release.sha256

# Then validate the chart tarball against the verified manifest:
sha256sum -c section-release.sha256
```

## Reproducing locally

Image builds are not bit-for-bit reproducible (we use multi-stage
Buildkit with cache mounts), but the **inputs** are pinned by
`pyproject.toml` + `uv.lock` and `pnpm-lock.yaml`. To approximate:

```bash
docker buildx build --platform linux/amd64 services/gateway -t local/gateway:repro
syft local/gateway:repro -o cyclonedx-json | diff - dist/sbom-gateway.cdx.json
```

A non-zero diff is expected on the build metadata (timestamps,
layer digests) but the package list should match.

## Static analysis and dependency hygiene

* **CodeQL** runs on every PR and weekly: `.github/workflows/codeql.yml`.
* **OpenSSF Scorecard** runs weekly and on branch-protection changes:
  `.github/workflows/scorecard.yml`.
* **Dependabot** opens grouped PRs weekly: `.github/dependabot.yml`.
* **Trivy / Grype** scanning of the published images is performed
  automatically by GHCR; results are visible under "Security" on the
  package page.

## Reporting weaknesses in the supply chain

Use GitHub PVR — see [`SECURITY.md`](../../SECURITY.md). Reports about
the build pipeline, signing identity, or attestation contents are
explicitly in-scope.
